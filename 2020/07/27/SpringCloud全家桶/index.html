Ribbon（停止维护，但依然大规模使用）

SpringCloud Ribbon是基于Netflix Ribbon实现的一套==客户端==负载均衡的工具，主要功能是提供客户端的软件负载均衡算法和服务调用（连接超时、重试等）。Ribbon会自动基于某种配置原则（轮询等）去连接这些机器。我们很容易使用Ribbon实现自定义的负载均衡算法。

网关篇：

Gateway：

概念：

SpringCloud Gateway作为SpringCloud生态系统中的网关，旨在替代Zuul，SpringCloud2.0以上的版本中，没有对新版本Zuul2.0以上最新高性能版本进行集成，仍然还是用的是Zuul1.x非Reactor模式的老版本。为了提升网关性能，SpringCloud Gateway是基于WebFlux框架实现的，而WebFlux框架底层则使用了高性能的Reactor模式通信框架Netty。

Gateway与Zuul的区别：

1、Zuul1.x 是一个基于阻塞I/O的API Gateway

2、Zuul1.x 基于Servlet2.5使用阻塞框架，他不支持任何长连接（如WebSocket），Zuul的设计模式和Nginx较像，每次I/O操作都是从工作线程中选一个执行，请求线程被阻塞到工作线程完成。但差别是Nginx用C++实现，Zuul用Java，JVM本身会有第一次加载较慢的情况，Zuul性能相对较差

3、Zuul2.x 的理念更加先进，但还是半成品

4、SpringCloud Gateway建立在Spring5、Project Reactor和SpringBoot 2之上，使用非阻塞API

5、SpringCloud Gateway还支持WebSocket，并且与Spring紧密集成拥有更好的开发体验

以上参考自

传统Web框架：struts、springmvc都是基于Servlet API和Servlet容器基础之上运行的

Servlet3.1之后有了异步非阻塞的支持，而WebFlux是一个典型的非阻塞异步的框架，核心是基于Reactor的相关API实现，可以运行在诸如Netty，Undertow及支持Servlet3.1的容器上。

1、Binder：连接中间件，屏蔽差异

2、Channel：通道，Queue的一种抽象，在消息通讯系统中实现存储和转发的媒介，通过Channel对Queue进行配置

3、Source & Sink：从Stream发送消息就是输出，接受消息就是输入

Provider -> Source -> Channel -> Binder -> MQ -> Binder -> Channel -> Sink -> Consumer

流控规则：

流控模式：

1、QPS直接 -> 快速失败

资源名：唯一名称，默认请求路径

狂点：

线程数限制和QPS的区别：

QPS不解释了，线程数假设是1，那就是一次允许一个线程来访问，如果该请求正在处理中，再来个线程访问就报错，换句话说，虽然实际情况连点不会导致报错——因为处理快，在controller那里添加一个sleep睡个1秒，连点就会报错。

2、关联

B达到阀值，A挂了 —— 背锅， 比如支付和下单接口，支付接口占用太多，就关闭下单接口

流控效果：

1、快速失败 —— 默认

2、预热

默认coldFactor —— 冷加载因子为3，即请求QPS从threshold/3开始，经郁热时长（自己设定）逐渐升至阀值

3、排队等待（只支持QPS）

匀速排队，严格控制请求通过的间隔时间，让请求匀速通过，对应的是漏统算法

用来处理间隔性突发的流量 —— 消息队列……我们希望在大量请求到来然后空闲的这段时间逐渐处理这些请求

熔断降级：

1、RT：平均响应时间，秒级

==平均响应时间超出阀值 && 在时间窗口时间内通过的请求 >= 5==，触发降级，窗口期过后关闭断路器。RT最大4900，更大的需要通过-Dcsp.sentinel.statistic.max.rt=XXXX才能生效

200ms以内必须处理完（时间窗口1s内请求>5），处理不完就开启断路器等待时间窗口时间到

Jmeter测试的时候开10/s的线程去访问，断路器打开，Jmeter关闭，时间到以后断路器关闭

2、异常比例：秒级

范围：0.1-1.0

QPS >= 5 且异常比例（秒级统计）超过阀值，触发降级；时间窗口结束后，关闭降级

换句话说单独访问必然是走正常流程

3、异常数：分钟级 —— 时间窗口 >= 60s

分钟内超过阀值，触发降级；后续同上

注：Sentinel断路器没有半开状态
