JVM 跨平台原理：

Java不直接和操作系统交互，而是通过JVM的解释，JVM根据不同平台解释成相应可以处理的机器码来执行 。

JVM种类（了解）：

  1. Hot Spot(我们用的)
  2. JRockit
  3. J9VM

JVM简单结构图：

1.类加载器：

  • 类加载器将编译后生成的*.class文件加载如JVM，并生成类的唯一标识Class类（可以理解为模板），Class类记录了类的构造器、方法名等信息（我们反射去到的类信息就是来自于该类，synchronize锁static方法的时候锁的也是该类）。

类加载器分类：

  1. 虚拟机自带的加载器
  2. 启动类（根）加载器——jre/lib/rt.jar包，现在好像没了。。
  3. 扩展类加载器——jre/lib/ext
  4. 应用程序类加载器
  5. 自定义类加载器——自己实现ClassLoader抽象类

双亲委派机制：

  1. 类加载器收到加载类的请求
  2. 将这个请求向上委托给父类加载器去完成，一直向上委托，直到启动类（根）加载器
  3. 启动加载器检查是否能够加载当前这个类，能加载就结束并使用当前的加载器，否则抛出异常，通知子类加载器进行加载
  4. 重复3
  5. 没找到就ClassNotFound

类加载器加载类的时候，加载器会一层一层往上找：如果我自己定义了一个String类，那加载器在应用程序层找到我的String类以后依然会继续往上层搜，如果搜到了，那我定义的String就会被无视。

代码中可以看出最后找到根加载器的时候显示是null，即java获取不到，因为这里是用了本地方法（java底层用C++实现）

2. 方法区

顺带提及：刚才提到了调用了C++写的本地方法，也就是native关键字，凡事带了native关键字的就是调用了C语言的库，会进入本地方法栈调用本地方法接口，也就是所谓的JNI来扩展java的使用

3. 程序计数器（PC寄存器）

每个线程独有一个程序计数器，线程私有，就是一个指针，指向方法区的方法字节码，占用的空间很小，可以忽略不计

4. 方法区：线程共享

方法区存有类的构造函数、接口，即类信息，以及静态变量、常量（所谓常量池）；换成关键字，就是static、final、Class模版类

注意：new出来的实例以及实例属性都存在堆内存中

5. 栈：Stack

栈主管程序运行，生命周期与线程同步——线程结束，栈内存释放

栈的存储内容：8大基本类型 + 对象引用地址 + 实例的方法（引用）

栈运行原理：元素单位——栈帧

线程每执行一个方法，就往栈内弹入一个栈帧；

换句话说，线程正在执行的方法一定在栈的顶部；

如果栈满了（A和B递归调用对方），就会报StackOverflow

栈帧组成：

  1. 方法索引
  2. 输入输出参数
  3. 本地变量
  4. Class File：引用 —— 指向堆内存中的实例
  5. 父帧 —— 指向调用者
  6. 子帧 —— 指向被调用者

栈不存在垃圾回收（可以这么理解，如果存在垃圾要等着回收，那栈既然是FILO，方法调用完如果顶部有垃圾等待回收，程序就会一直阻塞直到触发垃圾回收，那效率不就没了嘛）

堆：Heap

一个JVM只有一个堆内存，一个堆内存可以调节：VM option

类加载器读取文件后，把类、方法、常量、变量

堆内存分区：

  1. 新生区 ->轻GC

    • 新生区(Eden、伊甸区)
    • 幸存0区(Survive From)
    • 幸存1区(Survive To)

  2. 养老区（Old）->重GC/Full GC

  3. 元空间（以前叫永久区）

GC主要发生在新生区和养老区

堆内存满了：OOM；90%以上都是临时对象

元空间（常驻内存，存放jdk自身携带的Class对象，interface元数据，即储存运行的环境、类信息）：

不存在垃圾回收，关闭JVM就会释放这个区域的内存

jdk1.6之前：永久代，常量池在方法区

jdk1.7：永久代慢慢退化，提出 “去永久代”，常量池在堆中

jdk1.8之后：无永久代，常量池在元空间，元空间物理层面与堆无关，但逻辑可以理解为和堆在一起

可能出现的问题：一个启动类加载了大量第三方jar包、Tomcat部署了太多应用、大量动态生成的反射类。导致OOM

堆内存大小

默认情况下，分配的总内存是电脑内存的1/4，初始化内存是电脑的1/64（1/4 * 1/16）

调节堆内存默认值：-Xms1024m -Xmx1024m -XX:+PrintGCDetails

观察可以发现：PSYoungGen + ParOldGen $\approx$ 1024m（上面的配置配的）

也就是说：元空间逻辑上存在，物理上不存在堆

分析OOM和堆内存的工具：JProfile

GC常用算法：

  1. 引用计数法：记录引用的对象，将没有引用到的清除

  2. 复制算法：记住一句话——谁空谁是To，-XX:MaxTenuringThreshold=n设定进入老年代的时间。

    轻GC使Eden为空，留下的进入To（此时To即将不为空），From的也清理完以后进入现在的To，然后To和From交换名字，那To又是null了。

    • 优点：没有内存碎片
    • 缺点：浪费了内存空间，有一半幸存区永远是空To。

    最佳使用场景：对象存活度较低（新生区）

  3. 标记清除法：扫描对象，对引用的标记对象 ；再次扫描，清除为标记的对象

    • 优点：不需要额外空间
    • 两次扫描浪费时间，会产生内存碎片

  4. 标记压缩：

    标记清除执行数次的基础上进行内存压缩，多了一个移动成本

小结：

  1. 内存效率（时间复杂度）：复制算法 > 标记清除 > 标记压缩
  2. 内存整齐度：复制算法 = 标记压缩 > 标记清除
  3. 内存利用率：标记压缩 = 标记清除 > 复制算法

GC：分代收集算法：

新生代：存活率低，使用复制算法

老年代：区域大，存活率高，标记清除（内存碎片不是太多的时候） + 标记压缩
